================================================================================
1. EXECUTIVE SUMMARY
================================================================================

GetAWay is a browser-native AI-powered interview proctoring platform built on
Next.js 16, Firebase, and WebRTC. The platform targets a dual-user audience:
authenticated recruiters who create and monitor sessions, and unauthenticated
candidates who join via shared room links.

SECURITY POSTURE OVERVIEW
--------------------------
The application demonstrates a thoughtful but early-stage security approach.
The API layer correctly uses Firebase Admin SDK for token verification on
recruiter-facing routes, and the OTP system correctly hashes codes before
storage. However, several critical attack surfaces remain unprotected, most
notably: the candidate-facing API routes have zero authentication or rate
limiting, Firestore rules allow unrestricted writes to session subcollections,
and the recruiter dashboard has no middleware-enforced authentication guard.

MAIN STRENGTHS (What Exists Today)
------------------------------------
1. Firebase Admin SDK used for server-side token verification — recruiter API
   routes (rooms, analytics, settings) are properly protected.
2. OTP codes stored as SHA-256 hashes — plain text OTPs never touch Firestore.
3. OTPs marked as "used" after single consumption — replay attacks prevented.
4. OTP expiry enforced (10-minute TTL via Firestore field check).
5. Zod validation schemas applied to all API inputs — reduces injection risk.
6. otpCodes Firestore collection correctly blocked from all client access
   (allow read, write: if false).
7. Recruiters collection scoped by Firebase Auth UID — recruiter data isolation
   enforced at Firestore rule level.
8. Environment variables correctly separated (NEXT_PUBLIC_ for client,
   private for server-only keys).
9. .env* correctly gitignored — credentials not in source control.
10. Admin SDK gracefully degrades if credentials missing during build.
11. Structured API response helpers (apiSuccess, apiError, apiValidationError)
    ensure consistent error format.
12. React Strict Mode enabled — catches common development-time issues.

MAIN WEAKNESSES (Critical Gaps)
---------------------------------
1. NO MIDDLEWARE GUARD: The /recruiter/* pages have no server-side auth
   protection. Any user can access the recruiter dashboard URLs directly.
2. UNAUTHENTICATED WRITE APIS: Six critical API routes (alerts, trust logs,
   candidate onboarding, session start/end, report compilation) accept
   writes from anyone with a room ID — no authentication required.
3. WEAK ROOM ID ENTROPY: Room IDs follow format "GW-XXXX" (4-digit numeric
   suffix = only 9,000 possible values). Enumerable in under 1 minute.
4. FULLY OPEN FIRESTORE SUBCOLLECTIONS: alerts, trustLogs, report, and webrtc
   subcollections allow read and write from any unauthenticated source.
5. NO RATE LIMITING: OTP send/verify endpoints have no rate limiting — brute
   force and OTP flooding attacks are trivially possible.
6. MOCK STORE IN PRODUCTION PATH: The Zustand mockStore is used in recruiter
   pages (sessions, initialize, overview, report, settings) — state is not
   backed by Firestore and will be lost on page refresh or lost between users.
7. INTERVIEWER ROLE NOT PROTECTED: Any user who appends ?view=interviewer to
   a room URL gains the interviewer panel — no recruiter authentication check.
8. WEBHOOK URL NOT VALIDATED: The webhook URL stored in recruiter settings is
   used in a server-side fetch() call without domain whitelisting or SSRF
   protection.
9. NO CONTENT SECURITY POLICY (CSP): No security headers configured in
   next.config.ts or vercel.json.
10. FREE PUBLIC TURN SERVERS: openrelay.metered.ca is a public, unauthenticated
    TURN relay — candidate media traffic is routed through untrusted servers.
11. SENSITIVE DATA LEAKED IN ERRORS: error.tsx exposes raw error.message to
    the end user — internal stack traces may be revealed.

PRODUCTION READINESS SCORE: 3.5 / 10

The application is suitable for a controlled academic demonstration or MVP
prototype. It is NOT production-ready for real hiring workflows due to the
unprotected write endpoints, weak room ID entropy, and missing middleware
authentication.

2. SECURITY PRINCIPLES
================================================================================

The following principles should guide all security decisions for GetAWay.

1. LEAST PRIVILEGE
   ----------------
   Every user, component, and API route should have access ONLY to the
   resources it absolutely needs — nothing more.
   
   Current state: Recruiters have correct privilege scoping for their own data.
   Candidates and session participants have excessive write access to any room.
   Recommendation: Apply role-based access at API and Firestore rule level.

2. DEFENSE IN DEPTH
   ------------------
   Security should exist at multiple layers — even if one layer fails,
   another prevents damage. Do not rely on a single control.
   
   Current state: Token verification exists at API layer for recruiter routes.
   Firestore rules are the only guard for candidate/session writes, and they
   are fully open. There is no middleware layer, no IP filtering, and no
   anomaly detection.
   Recommendation: Add middleware auth, tighten Firestore rules, and add rate
   limiting as three independent protective layers.

3. SECURE BY DEFAULT
   -------------------
   New features and routes should default to the most restrictive access
   possible. Access must be explicitly granted, not implicitly allowed.
   
   Current state: New room subcollections default to open access. The WebRTC
   signaling collection is fully open.
   Recommendation: All new Firestore paths should default to "allow: if false"
   until access requirements are explicitly defined and implemented.

4. PRIVACY BY DESIGN
   -------------------
   Candidate personal data (name, email, role, behavioral data from AI
   proctoring) should be collected with consent, stored minimally, and
   protected appropriately.
   
   Current state: Consent checkbox is present in RulesStep. Candidate email
   and name are stored but not encrypted at rest. There is no data retention
   policy or deletion mechanism.
   Recommendation: Add data retention policies, encrypted storage for PII,
   and a candidate data deletion pathway.

5. ZERO TRUST
   -----------
   Never assume that because traffic is coming from inside the system, it is
   safe. Verify every request regardless of origin.
   
   Current state: The session-related API routes (alerts, trust, candidate)
   trust any caller with a valid room ID. Room IDs are treated as secrets but
   are too short to be effective.
   Recommendation: Issue short-lived session tokens to verified session
   participants. Verify these tokens on every write request.

6. FAIL SECURE
   ------------
   When the system encounters an error, it should default to denying access
   rather than granting it.
   
   Current state: The Admin SDK initialization includes a graceful fallback
   (makeMissingProxy) which prevents hard crashes. API routes return generic
   500 errors on failure.
   Recommendation: Ensure all critical paths fail with access denied (403/401)
   rather than exposing internal error details.

3. THREAT MODEL
================================================================================

ASSETS (What We Are Protecting)
---------------------------------
- Recruiter account credentials (email/password in Firebase Auth)
- Candidate personal data (name, email, role — stored in Firestore rooms/{id})
- Session behavioral data (trust logs, AI alerts — audit-sensitive)
- Post-session reports (contains integrity scores — employment-sensitive)
- SMTP credentials (SMTP_USER, SMTP_PASS — email delivery capability)
- Firebase Admin private key (server-side — full Firestore access)
- Webhook URLs (recruiter-configured — SSRF attack surface)
- WebRTC media streams (live audio/video — highly private)

SENSITIVE DATA
---------------
- Firebase Admin private key (FIREBASE_ADMIN_PRIVATE_KEY) — CRITICAL
- SMTP_PASS — HIGH
- Candidate email addresses — HIGH (PII)
- Candidate names — MEDIUM (PII)
- Behavioral/trust score data — HIGH (employment decision data)
- OTP codes (hashed at rest — protected)

ATTACK SURFACES
---------------
1. Public interview page  (/interview/[roomId])
2. OTP send/verify API    (/api/auth/otp/send, /api/auth/otp/verify)
3. Candidate API          (/api/rooms/[roomId]/candidate)
4. Trust log API          (/api/rooms/[roomId]/trust)
5. Alerts API             (/api/rooms/[roomId]/alerts)
6. Session start/end API  (/api/rooms/[roomId]/session/start|end)
7. Report compile API     (/api/rooms/[roomId]/report)
8. WebRTC signaling       (Firestore /webrtc/{roomId}/* collections)
9. Webhook endpoint       (outbound fetch in session/end/route.ts)
10. Recruiter dashboard   (/recruiter/* — no middleware guard)
11. TURN server           (openrelay.metered.ca — public relay)

THREAT ANALYSIS
----------------

THREAT 1: ROOM ID ENUMERATION
Surface: GET /interview/[roomId], all /api/rooms/[roomId]/* routes
Risk Level: CRITICAL
Impact: Attacker enumerates valid room IDs (only 9,000 possibilities with
        format GW-XXXX), accesses live sessions of other candidates, reads
        or tampers with trust scores and reports.
Current Mitigation: None — room IDs are sequential 4-digit numbers.
Required Mitigation: Change room ID to cryptographically random UUID or 128-bit
                     token. Example: GW-a3f9e2b1c4d7.

THREAT 2: UNAUTHENTICATED TRUST SCORE MANIPULATION
Surface: POST /api/rooms/[roomId]/trust
Risk Level: CRITICAL
Impact: Any person on the internet can POST arbitrary trust scores to any room
        by guessing or brute-forcing a room ID. A candidate could manipulate
        their own trust score to 100% throughout the session.
Current Mitigation: None.
Required Mitigation: Require session token or IP binding. At minimum, validate
                     caller identity against the room's expected participants.

THREAT 3: UNAUTHENTICATED ALERT INJECTION
Surface: POST /api/rooms/[roomId]/alerts
Risk Level: HIGH
Impact: An attacker can inject false alerts ("Multiple persons detected") into
        any candidate's session, poisoning the post-session report and
        potentially causing wrongful rejection of a candidate.
Current Mitigation: Zod schema validation exists (message, issuedBy fields).
Required Mitigation: Require authentication. Restrict issuedBy=INTERVIEWER to
                     verified recruiters only.

THREAT 4: OTP BRUTE FORCE
Surface: POST /api/auth/otp/verify
Risk Level: HIGH
Impact: A 6-digit OTP has only 1,000,000 possible values. Without rate
        limiting, an attacker can try all combinations in seconds to take
        over any recruiter account.
Current Mitigation: 10-minute OTP expiry, single-use enforcement.
Required Mitigation: Implement rate limiting (max 5 attempts per email per
                     10 minutes). Lock account after threshold breach.

THREAT 5: INTERVIEWER IMPERSONATION
Surface: /interview/[roomId]?view=interviewer
Risk Level: HIGH
Impact: ANY person with a room link can append ?view=interviewer to their URL
        and gain the full interviewer panel — including the ability to issue
        warnings and view live trust scores.
Current Mitigation: None.
Required Mitigation: Verify Firebase Auth token when view=interviewer is
                     requested. Confirm the authenticated UID matches the
                     room's recruiterId.

THREAT 6: WEBHOOK SSRF (Server-Side Request Forgery)
Surface: POST /api/rooms/[roomId]/session/end (webhook fire)
Risk Level: HIGH
Impact: A recruiter (or attacker who controls recruiter settings) can configure
        a webhook URL pointing to internal services: http://169.254.169.254
        (AWS metadata), http://localhost:*, or other internal endpoints.
        The server-side fetch() will execute against these internal targets.
Current Mitigation: None — URL stored and fetched without validation.
Required Mitigation: Validate webhook URL against allowlist of external
                     domains. Block RFC1918 addresses and loopback IPs.

THREAT 7: CANDIDATE DATA EXPOSURE VIA ROOM ID
Surface: GET /api/rooms/[roomId]/candidate, Firestore rooms/{id}
Risk Level: MEDIUM
Impact: Anyone who obtains a room ID can read the candidate's full name,
        email address, and the role they applied for.
Current Mitigation: None.
Required Mitigation: Scope candidate data access to authenticated recruiters
                     owning the room.

THREAT 8: SESSION TAKEOVER (Replay Attack)
Surface: /interview/[roomId] (WebRTC join)
Risk Level: MEDIUM
Impact: A candidate link, once shared or intercepted, can be reused by a
        third party after the legitimate candidate has joined or the session
        has ended.
Current Mitigation: Room status is tracked (OPEN/ACTIVE/ENDED) but not
                     enforced at the join level.
Required Mitigation: Enforce room status checks before allowing new WebRTC
                     connections. Expire links after session ends.

THREAT 9: FIRESTORE SIGNALING DATA EXPOSURE
Surface: /webrtc/{roomId}/signal, callerCandidates, calleeCandidates
Risk Level: MEDIUM
Impact: SDP (Session Description Protocol) offers and ICE candidates are
        stored in Firestore with fully open read access. An attacker who
        obtains a room ID can read SDP data and potentially gather network
        topology information.
Current Mitigation: Data is transient and cleaned up on hangup.
Required Mitigation: Apply TTL policies. Consider server-side signaling relay.

THREAT 10: SMTP CREDENTIAL EXPOSURE
Surface: Vercel environment variables, local .env.local
Risk Level: MEDIUM
Impact: If Vercel environment settings are compromised, SMTP credentials
        (SMTP_USER, SMTP_PASS) could be used to send spam or phishing
        emails at scale using the platform's email identity.
Current Mitigation: .env* is gitignored. Credentials stored in Vercel.
Required Mitigation: Use app-specific passwords or OAuth2 for Gmail SMTP.
                     Rotate credentials quarterly.

4. USER ROLES & ACCESS MATRIX
================================================================================

IDENTIFIED ROLES
-----------------

ROLE 1: RECRUITER (Authenticated)
   Authentication: Firebase Auth (email/password)
   Identifier: Firebase Auth UID

   CAN DO:
   - Create new interview rooms (POST /api/rooms)
   - List their own rooms (GET /api/rooms)
   - View session reports for their rooms
   - Configure AI thresholds and settings (GET/PUT /api/settings)
   - View analytics for their account (GET /api/analytics)
   - Join interview as interviewer (/interview/[roomId]?view=interviewer)
   - Issue manual warnings during sessions
   - Configure webhook URL for session notifications

   CANNOT DO (by design):
   - Access other recruiters' rooms or data
   - Read/write other recruiters' settings
   - Modify otpCodes collection directly

   SENSITIVE ACTIONS:
   - Password reset via OTP
   - Webhook URL configuration (SSRF risk)
   - Deleting rooms (requires auth + ownership)

ROLE 2: CANDIDATE (Unauthenticated)
   Authentication: None (room ID acts as credential)
   Identifier: Room ID + onboarding form data

   CAN DO:
   - Access interview room page via room link
   - Submit onboarding information (name, email, role)
   - Provide consent
   - Join WebRTC session as candidate
   - Toggle their own camera and microphone

   CANNOT DO:
   - Access recruiter dashboard
   - View other candidates' data
   - Modify trust scores or alerts directly (by design, these are AI-only)
   - View post-session reports

   SENSITIVE ACTIONS:
   - Accepting recording consent (legal requirement)
   - Sharing personal data (name, email)

ROLE 3: SYSTEM / AI PROCTORING ENGINE (In-Browser Service)
   Authentication: Operates within authenticated candidate session
   Identifier: Room ID

   CAN DO:
   - POST trust score updates (/api/rooms/[roomId]/trust)
   - POST alerts for detected violations (/api/rooms/[roomId]/alerts)
   - Trigger session start/end (/api/rooms/[roomId]/session/start|end)
   - Compile post-session report (POST /api/rooms/[roomId]/report)

   SENSITIVE ACTIONS:
   - All proctoring writes (currently unprotected)

ROLE 4: ADMINISTRATOR (Not yet implemented)
   This role does not currently exist in the codebase.
   [Recommendation] A future admin role should be able to:
   - View all recruiter accounts
   - Suspend accounts
   - Access platform-wide analytics
   - Manage SMTP and Firebase configurations

AUTHENTICATION PROVIDERS
------------------------
- Firebase Authentication (Email/Password) — Recruiter login/signup
- OTP via Nodemailer/SMTP — Recruiter password reset
- No authentication — Candidates (room ID is the credential)

RECRUITER LOGIN FLOW
---------------------
[Browser]                    [Next.js API]              [Firebase Auth]
    |                              |                           |
    | 1. Submit email + password   |                           |
    |----------------------------> |                           |
    |                              | 2. getAuth().signInWithEmailAndPassword()
    |                              |--------------------------->|
    |                              |                           | 3. Validate credentials
    |                              | <-------------------------|
    |                              | 4. Return ID token        |
    | <--------------------------- |                           |
    | 5. Store token in SDK        |                           |
    | 6. Future requests: Bearer <token>                       |
    |----------------------------> |                           |
    |                              | 7. adminAuth.verifyIdToken(token)
    |                              |--------------------------->|
    |                              |                           | 8. Decode + return uid
    |                              | <-------------------------|
    |                              | 9. Scope query to uid     |
    | 10. Return response          |                           |
    | <--------------------------- |                           |

SECURITY STRENGTHS:
- Firebase ID tokens are short-lived JWTs (1-hour expiry by default)
- Token verification uses Firebase Admin SDK (cryptographic validation)
- uid extracted from verified token ensures data isolation

SECURITY WEAKNESSES:
- No middleware to enforce auth on /recruiter/* page routes
- Token is managed by Firebase SDK — no custom session duration control
- No refresh token rotation policy implemented
- No multi-factor authentication

OTP FLOW (Password Reset)
--------------------------
[Browser]                  [Next.js API]              [Firestore]          [Email]
    |                           |                          |                   |
    | 1. POST /api/auth/otp/send {email}
    |-------------------------> |                          |                   |
    |                           | 2. Generate 6-digit OTP (Math.random)
    |                           | 3. SHA-256 hash OTP      |                   |
    |                           | 4. Store: otpCodes/{email} = {hash, expiresAt, used:false}
    |                           |------------------------->|                   |
    |                           | 5. sendOtpEmail(email, plainOTP)
    |                           |---------------------------------------------------->|
    | 6. Return {message: "OTP sent"}
    | <------------------------ |                          |                   |
    |                           |                          |                   |
    | 7. POST /api/auth/otp/verify {email, otp, newPassword}
    |-------------------------> |                          |                   |
    |                           | 8. Fetch otpCodes/{email}
    |                           |<-------------------------|                   |
    |                           | 9. Check: used == false  |                   |
    |                           | 10. Check: now < expiresAt                  |
    |                           | 11. SHA-256(otp) == stored hash             |
    |                           | 12. Mark used = true     |                   |
    |                           |------------------------->|                   |
    |                           | 13. adminAuth.getUserByEmail(email)         |
    |                           | 14. adminAuth.updateUser(uid, {password: newPassword})
    |                           | 15. adminAuth.createCustomToken(uid)        |
    | 16. Return {customToken}  |                          |                   |
    | <------------------------ |                          |                   |
    | 17. signInWithCustomToken(customToken)               |                   |

SECURITY STRENGTHS:
- OTP is hashed with SHA-256 before storage (never plain text in Firestore)
- OTP is marked as "used" after consumption — replay prevention
- OTP expires in 10 minutes
- New password sent in body (not URL params) — no log exposure

SECURITY WEAKNESSES:
- OTP generated with Math.random() — NOT cryptographically random
  (Should use crypto.randomBytes() or crypto.getRandomValues())
- No rate limiting on OTP send or verify endpoints
- No account lockout after failed OTP attempts
- Email subject says "Password Reset Code" — could be phishing vector
  if domain not verified

6. AUTHORIZATION ARCHITECTURE
================================================================================

ACCESS CONTROL MODEL
---------------------
GetAWay uses a hybrid access control approach:

1. API-Level: Token-based identity verification (Firebase JWT)
   - Applied to: rooms (CRUD), analytics, settings
   - Method: verifyAuthToken() in src/lib/api.ts

2. "Security by Obscurity" (Unguessable Room ID)
   - Applied to: candidate and session-related API routes
   - Risk: Room IDs are only 9,000 possibilities — NOT truly unguessable
   
3. Firestore Rules: Database-level enforcement
   - Applied to: all Firestore collections
   - Problem: Subcollections are fully open

OWNERSHIP MODEL
----------------
- Rooms: Owned by recruiterId (Firebase UID) stored in room document
- Recruiter data: Owned by document ID matching Firebase UID
- Session data: Associated to room via room document ID
- OTP codes: Server-managed, no client ownership

RESOURCE ACCESS — CURRENT STATE
---------------------------------

ROOMS: WHO CAN ACCESS
  Read: Anyone with the room ID (Firestore allow read: if true)
  Create: Authenticated recruiter (via API + Firestore rule)
  Update: Anyone (allow update: if true — needed for candidate onboarding)
  Delete: Authenticated recruiter (API checks auth, Firestore checks uid)

REPORTS: WHO CAN ACCESS
  Read: Anyone who knows the room ID (POST /api/rooms/[id]/report has no auth)
  Write: Anyone who knows the room ID (no auth check on compile endpoint)
  Risk: Report can be overwritten by attacker with room ID

ALERTS: WHO CAN ACCESS
  Read: Anyone (no auth)
  Write: Anyone (no auth) — attacker can inject false alerts

ANALYTICS: WHO CAN ACCESS
  Read: Authenticated recruiter (scoped to their own rooms by recruiterId)
  Security: GOOD — correctly protected

SETTINGS: WHO CAN ACCESS
  Read/Write: Authenticated recruiter (scoped by UID)
  Security: GOOD — correctly protected

WEBRTC SIGNALING: WHO CAN ACCESS
  Read/Write: Anyone (fully open in Firestore rules)
  Rationale: Room IDs are "unguessable" but only 9,000 possibilities
  Risk: SDP hijacking if room ID is compromised

7. FIRESTORE SECURITY REVIEW
================================================================================

COLLECTION: recruiters/{uid}
  Purpose: Recruiter profile and settings storage
  Who can read: Only the recruiter matching the document UID
  Who can write: Only the recruiter matching the document UID
  Who should NOT access: Other recruiters, candidates, public
  Risk Level: LOW (correctly secured)
  Current Rule: allow read, write: if request.auth != null && request.auth.uid == uid
  Assessment: ADEQUATE for MVP. Production should add field-level validation.

COLLECTION: rooms/{roomId}
  Purpose: Interview room metadata and session state
  Who can read: EVERYONE (allow read: if true)

RECOMMENDED PRODUCTION-READY FIRESTORE RULES
---------------------------------------------

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // ─── Helper functions ───────────────────────────────────────────────
    function isAuthenticated() {
      return request.auth != null;
    }

    function isOwner(uid) {
      return isAuthenticated() && request.auth.uid == uid;
    }

    function roomExists(roomId) {
      return exists(/databases/$(database)/documents/rooms/$(roomId));
    }

    function isRoomOwner(roomId) {
      return isAuthenticated() &&
        get(/databases/$(database)/documents/rooms/$(roomId)).data.recruiterId
          == request.auth.uid;
    }

    // ─── Recruiter profiles ─────────────────────────────────────────────
    match /recruiters/{uid} {
      allow read: if isOwner(uid);
      allow write: if isOwner(uid);
    }

    // ─── Interview rooms ─────────────────────────────────────────────────
    match /rooms/{roomId} {
      // Recruiters can read their own rooms; allow public read only of non-PII
      allow read: if isRoomOwner(roomId) || (
        // [Recommendation] Implement a field mask for public candidate access
        resource.data.status == 'OPEN'
      );
      allow create: if isAuthenticated();
      // Only update allowed: candidate joining an open room (limited fields)
      allow update: if isRoomOwner(roomId) ||
        // Candidate updates: only allowed if room is OPEN and only candidate field
        (resource.data.status == 'OPEN' &&
         request.resource.data.diff(resource.data).affectedKeys()
           .hasOnly(['candidate', 'status']));
      allow delete: if isRoomOwner(roomId);

      // Alerts: only room owner can read; AI system (via Admin SDK) writes
      match /alerts/{alertId} {
        allow read: if isRoomOwner(roomId);
        allow write: if false; // Server-only via Admin SDK
      }

      // Trust logs: only room owner can read; AI system (via Admin SDK) writes
      match /trustLogs/{logId} {
        allow read: if isRoomOwner(roomId);
        allow write: if false; // Server-only via Admin SDK
      }

      // Report: only room owner can read; compiled server-side
      match /report/{doc} {
        allow read: if isRoomOwner(roomId);
        allow write: if false; // Server-only via Admin SDK
      }
    }

    // ─── WebRTC signaling ────────────────────────────────────────────────
    // [Recommendation] Add session token validation when implemented
    match /webrtc/{roomId} {
      match /signal/{doc} {
        // Only allow read/write if the parent room exists and is ACTIVE
        allow read, write: if roomExists(roomId) &&
          get(/databases/$(database)/documents/rooms/$(roomId)).data.status
            in ['OPEN', 'ACTIVE'];
      }
      match /callerCandidates/{doc} {
        allow read, write: if roomExists(roomId);
      }
      match /calleeCandidates/{doc} {
        allow read, write: if roomExists(roomId);
      }
    }

    // ─── OTP codes ────────────────────────────────────────────────────────
    match /otpCodes/{email} {
      allow read, write: if false; // Server-only via Admin SDK
    }
  }
}

RECRUITERS COLLECTION
  Access Policy: A recruiter document can ONLY be read or written by the
  recruiter whose Firebase UID matches the document ID.
  
  Plain English: "You can only see and edit your own account profile."
  
  Ownership Rule: Document ID = Firebase Auth UID
  Implementation: Firestore rule "request.auth.uid == uid" — CORRECTLY DONE
  Gap: No field-level validation prevents a recruiter from writing invalid
       data types (e.g., setting aiThreshold to -999).

ROOMS COLLECTION
  Access Policy (CURRENT — Insecure):
    Anyone can read any room. Anyone can update any room.
  
  Access Policy (RECOMMENDED):
    A room can be READ by its owner recruiter only.
    A room can be CREATED by any authenticated user.
    A room can be UPDATED by its owner, or by a candidate (restricted fields only).
    A room can be DELETED by its owner only.
  
  Ownership Rule: rooms/{roomId}.data.recruiterId = recruiter's Firebase UID
  
  Plain English: "Each recruiter sees only their own interview rooms.
  Candidates can only update the candidate field on an open room."

ALERTS SUBCOLLECTION
  Access Policy (CURRENT — Insecure): Anyone can read and write alerts.
  
  Access Policy (RECOMMENDED):
    Alerts can only be WRITTEN by the server via Admin SDK.
    Alerts can only be READ by the recruiter who owns the room.
  
  Plain English: "Only the AI system (running on the server) can create alerts.
  Only the recruiter can review them."
  
  Why This Matters: False alerts could wrongly disqualify a candidate.

TRUSTLOGS SUBCOLLECTION
  Access Policy (CURRENT — Insecure): Anyone can read and write trust logs.
  
  Access Policy (RECOMMENDED):
    TrustLogs can only be APPENDED (never updated) by the server via Admin SDK.
    TrustLogs can only be READ by the recruiter who owns the room.
  
  Plain English: "Trust scores are an immutable audit trail. Only the
  AI proctoring server can add to it. Only the recruiter can read it."

REPORT SUBCOLLECTION
  Access Policy (CURRENT — Insecure): Anyone can read and write reports.
  
  Access Policy (RECOMMENDED):
    Reports can only be WRITTEN by the server at session end.
    Reports can only be READ by the recruiter who owns the room.
  
  Plain English: "Interview reports are private to the recruiter. They cannot
  be modified by candidates or third parties."

ANALYTICS
  Access Policy (CURRENT): API route correctly scopes to recruiterId.
  Policy: Recruiter can only view analytics for their own rooms.
  Status: API level is correctly secured. No Firestore rule needed (read via API).

WEBRTC SIGNALING
  Access Policy (CURRENT — Insecure): Fully open.
  
  Access Policy (RECOMMENDED):
    Read/write only allowed when the parent room document exists and is in
    OPEN or ACTIVE status.
  
  Plain English: "WebRTC signaling only works for rooms that are currently
  active. Once a session ends, signaling data becomes inaccessible."

OTPCODES COLLECTION
  Access Policy: Server-only. Client reads and writes are blocked.
  Status: CORRECTLY IMPLEMENTED — allow read, write: if false.

10. CANDIDATE ACCESS SECURITY
================================================================================

PUBLIC ROOM LINKS
------------------
Room links follow the format: https://get-a-way.vercel.app/interview/GW-XXXX

The room ID is the ONLY security control for candidate access. This design is
intentional to minimize friction for candidates, but creates several risks.

ENUMERATION RISKS
  The current room ID format is "GW-" followed by a 4-digit number (1000-9999).
  This creates only 9,000 possible values. An attacker can enumerate all valid
  rooms in approximately:
    - 9,000 HTTP requests to /api/rooms/[roomId]/candidate
    - At 10 requests/second: 15 minutes to enumerate all rooms
    - At 100 requests/second: 90 seconds to enumerate all rooms
  
  Impact: An attacker can discover all active interview rooms, read candidate
  names and emails, and inject data into active sessions.
  
  [Recommendation] Change room ID to a UUID v4 or 128-bit random token.
  Example: GW-a3f9e2b1c4d7-8e9f-4a2b-c1d3 (2^128 possibilities)

LINK SHARING RISKS
  1. Room link forwarded to third party who joins as candidate
  2. Room link shared on social media (e.g., screenshot with URL visible)
  3. Room link intercepted in corporate email systems
  
  [Recommendation] Issue time-limited OTP links for candidates too. The link
  expires after the session ends or 24 hours, whichever is sooner.

REPLAY RISKS
  Once a session ends, the room URL remains valid. Anyone with the old link
  can still visit the URL and attempt to onboard as a candidate.
  
  Current behavior: No check in OnboardingStep or CandidateFlow to verify
  room status (OPEN/ACTIVE/ENDED) before proceeding.
  
  [Recommendation] Check room status at page load. If room.status == "ENDED",
  redirect to an expired-link page.

CONSENT COLLECTION
  The RulesStep component (src/components/interview/RulesStep.tsx) requires
  candidates to check a consent checkbox before entering the live room.
  
  Consent text: "I consent to being recorded and agree to follow all guidelines."
  
  ISSUES:
  1. Consent is a UI-only control — it is not recorded or persisted to Firestore.
  2. No timestamp of consent is stored.
  3. The legal name of the policy is not shown (no link to full privacy policy).
  4. The CandidateSubmitSchema does have a consentGiven: z.literal(true) field
     but the OnboardingStep currently calls mockStore (not the real API).
  
  [Recommendation]:
  1. Persist consent with timestamp to Firestore when candidate submits form.
  2. Include recruiter company name and session purpose in consent text.
  3. Link to full privacy policy before consent checkbox.
  4. Store: { consentGiven: true, consentTimestamp: ISO8601, ipAddress: string }

SESSION JOINING SECURITY
  The CandidateFlow component progresses candidates through three steps:
  Onboarding → Rules → Live Room. There is no server-side validation that:
  1. The room exists and is in OPEN status
  2. No other candidate has already joined
  3. The session has not already ended
  
  This allows multiple candidates to join the same room simultaneously,
  which would result in a corrupted WebRTC session and mixed proctoring data.

11. WEBRTC SECURITY REVIEW
================================================================================

SIGNALING ARCHITECTURE
------------------------
WebRTC signaling is handled via Firestore in a dedicated collection:
  /webrtc/{roomId}/signal/data        — SDP offer/answer
  /webrtc/{roomId}/callerCandidates   — Interviewer ICE candidates
  /webrtc/{roomId}/calleeCandidates   — Candidate ICE candidates

Implementation file: src/hooks/useWebRTC.ts (412 lines)

ICE / STUN Configuration:
  - stun.l.google.com:19302 (and stun1/stun2/stun3)
  - turn:openrelay.metered.ca:80 (PUBLIC, UNAUTHENTICATED)
  - turn:openrelay.metered.ca:443 (PUBLIC, UNAUTHENTICATED)
  - turn:openrelay.metered.ca:443?transport=tcp

SDP EXCHANGE FLOW
  Interviewer creates offer → writes to Firestore signal/data
  Candidate listens → receives offer → creates answer → writes to Firestore
  Both parties exchange ICE candidates → P2P connection established
  All audio/video transmitted P2P (SRTP encrypted by WebRTC standard)

CURRENT RISKS

1. PUBLIC TURN SERVER — CRITICAL
   The TURN relay used (openrelay.metered.ca) is a free public service.
   Credentials are hardcoded and publicly known:
     username: "openrelayproject"
     credential: "openrelayproject"
   
   Risks:
   - Any third party can use the same TURN relay with the same credentials
   - The relay operator can potentially intercept or log relayed traffic
   - TURN relay is a trusted intermediary for media when P2P fails
   - Service reliability is not guaranteed (can go down any time)
   
   [Recommendation] Deploy a dedicated TURN server (e.g., Coturn on a VPS)
   or use a paid TURN service (Twilio TURN, Cloudflare Calls, Metered.ca paid).
   Use time-limited TURN credentials generated per-session.

2. OPEN SIGNALING COLLECTION — HIGH
   The /webrtc/{roomId}/* collections are fully open. An attacker with a
   room ID can:
   - Read SDP offers/answers (reveals network topology)
   - Read ICE candidates (reveals local and reflexive IP addresses)
   - Write a fake SDP offer to disrupt the signaling process
   - Perform man-in-the-middle against the signaling channel

   [Recommendation] Restrict signaling read/write to participants with
   a valid session token. At minimum, gate on room existence and ACTIVE status.

3. METADATA EXPOSURE — MEDIUM
   ICE candidates stored in Firestore contain:
   - Local IP addresses (private network info)
   - Server-reflexive IP addresses (public IP of the user)
   - Port numbers in use
   
   These are visible to anyone who can read the callerCandidates collection.
   
   [Recommendation] Configure mdns candidates for local IP concealment.
   Apply proper Firestore rules to limit callerCandidates/calleeCandidates reads.

4. NO SIGNALING AUTH — HIGH
   The useWebRTC hook writes directly to Firestore using the Firebase Client SDK.
   There is no server-side validation that:
   - The caller is the actual interviewer for this room
   - The caller is the actual candidate for this room
   
   [Recommendation] Issue short-lived signed tokens to room participants.
   Verify tokens in Firestore rules before allowing signaling writes.

5. STALE SIGNAL DATA — MEDIUM
   The interviewer's cleanupSignal() function deletes signal data on leave.
   However, if the interviewer disconnects abruptly (browser crash, network loss),
   stale SDP data persists in Firestore indefinitely.
   
   [Recommendation] Implement Firestore TTL (Time-to-Live) policy on the
   webrtc collection. Delete signal data 2 hours after createdAt.

6. DEBUG LOGGING — LOW
   The useWebRTC hook logs extensively to console:
   "[WebRTC] Offer written to Firestore at webrtc/GW-XXXX/signal/data"
   "[WebRTC][Interviewer] Received callee ICE candidate"
   
   These logs expose signaling paths and room IDs in browser consoles.
   
   [Recommendation] Remove verbose console.log statements in production.
   Replace with a structured logging service that can be disabled.

MEDIA TRANSMISSION SECURITY
  - WebRTC uses DTLS-SRTP by default — media is encrypted in transit
  - Browser enforces DTLS handshake — keys are not exposed to JS code
  - Media never touches Vercel servers or Firestore (it is P2P)
  - Assessment: Media encryption is GOOD by default (WebRTC standard)

- Candidate name, email, role in rooms/{roomId}
- All trust logs in rooms/{roomId}/trustLogs/*
- All alerts in rooms/{roomId}/alerts/*
- Compiled report in rooms/{roomId}/report/compiled
- Session timestamps

[Recommendation] Implement a data retention policy:
- Candidate PII: 90 days after session end
- Trust logs: 1 year (audit purposes)
- Reports: 2 years (employment records compliance)
- OTP codes: Already auto-expired (10 minutes)

ENCRYPTION REQUIREMENTS
  Currently:
  - Firebase Auth stores passwords hashed (managed by Firebase)
  - Firestore data is encrypted at rest by Google (AES-256) — infrastructure level
  - Firestore data is encrypted in transit (TLS) — infrastructure level
  - OTP codes stored as SHA-256 hashes (application level)
  
  Missing:
  - Application-level encryption for candidate PII
  - Candidate email and name are stored in plain text in Firestore

  [Recommendation] Encrypt candidate email with AES-256 using a KMS-managed
  key before storing in Firestore. Decrypt only when needed by authorized
  recruiter requests.

PRIVACY CONSIDERATIONS
  1. Candidates are subject to real-time behavioral monitoring (face detection,

  3. There is no right-to-deletion mechanism for candidates.
  
  4. There is no privacy policy document linked from the application.
  
  [Recommendation] Implement GDPR/DPDP compliance features:
  - Right to access: candidates can request their data
  - Right to deletion: candidates can request erasure
  - Privacy policy linked from consent form
  - Data processing agreement for recruiters

14. ERROR HANDLING STRATEGY
================================================================================

CURRENT ERROR HANDLING IMPLEMENTATION
  - src/lib/api.ts: Consistent apiSuccess/apiError/apiValidationError helpers
  - src/app/error.tsx: Next.js error boundary (client-side)
  - src/components/GlobalErrorCatcher.tsx: window.onerror + unhandledrejection
  - API routes: try/catch blocks with console.error logging

COMPLETE ERROR HANDLING GUIDE
-------------------------------

ERROR: AUTHENTICATION FAILURE (401)
  Cause: Missing or invalid Firebase ID token in Authorization header
  User-facing message: "Your session has expired. Please sign in again."
  Internal log: "AUTH_FAILURE: verifyIdToken failed for request to {route}"
  Severity: MEDIUM
  Recovery: Redirect user to login page. Clear stale token from SDK.

ERROR: AUTHORIZATION FAILURE (403)
  Cause: Authenticated user attempting to access another user's resource
  User-facing message: "You don't have permission to access this resource."
  Internal log: "AUTHZ_FAILURE: uid={uid} attempted access to room={roomId}"
  Severity: HIGH (potential malicious intent)
  Recovery: Return 403. Log the attempt for security monitoring.

ERROR: VALIDATION FAILURE (422)
  Cause: Input data fails Zod schema validation
  User-facing message: "Invalid data submitted. Please check your input."
  Internal log: Zod error flatten (field-level errors)
  Severity: LOW
  Recovery: Return structured validation errors. Do not expose schema details.

ERROR: OTP EXPIRED (400)
  Cause: OTP submitted after 10-minute window
  User-facing message: "This code has expired. Please request a new one."
  Internal log: "OTP_EXPIRED: email={email}, expiredAt={timestamp}"
  Severity: LOW
  Recovery: Guide user to re-trigger OTP send.

ERROR: OTP ALREADY USED (400)
  Cause: OTP submitted a second time after already consumed
  User-facing message: "This code has already been used."
  Internal log: "OTP_REPLAY: email={email}, attempted at {timestamp}"
  Severity: MEDIUM (possible replay attack)
  Recovery: Return 400. Alert security team if repeated replay attempts.

ERROR: INVALID OTP (400)
  Cause: OTP hash does not match stored hash
  User-facing message: "Invalid code. Please check and try again."
  Internal log: "OTP_MISMATCH: email={email}, attempt={n}"
  Severity: HIGH (after 3+ attempts — possible brute force)
  Recovery: Increment attempt counter. Lock after 5 attempts.

ERROR: FIRESTORE UNAVAILABLE (503)
  Cause: Firebase/Firestore service outage
  User-facing message: "Service temporarily unavailable. Please try again in a moment."
  Internal log: "FIRESTORE_ERROR: {error.code} {error.message}"
  Severity: CRITICAL
  Recovery: Return 503. Implement retry with exponential backoff in critical paths.

ERROR: WEBRTC NEGOTIATION FAILURE (Client)
  Cause: ICE negotiation failure, network issue, TURN server unavailable
  User-facing message: "Could not connect. Try refreshing or switching networks."
  Internal log: Console.error in useWebRTC (currently verbose debug logging)
  Severity: MEDIUM
  Recovery: Show retry button. Cleanup stale Firestore signal data.

ERROR: NETWORK FAILURE (Client)
  Cause: User's internet connection dropped
  User-facing message: "Connection lost. Please check your network."
  Severity: LOW
  Recovery: WebRTC handles reconnection automatically for minor drops.

ERROR: WEBRTC MEDIA DENIED (Client — "denied" state)
  Cause: User denied camera/microphone browser permission
  User-facing message: "Camera and microphone access required. Please allow in browser settings."
  Severity: LOW
  Recovery: Show browser-specific permission instructions. Retry button.

ERROR: OTP EMAIL SEND FAILURE (500)
  Cause: SMTP server unavailable, invalid credentials, email bounce
  User-facing message: "Failed to send the code. Please try again."
  Internal log: "SMTP_ERROR: {error.message} for email={email}"
  Severity: HIGH
  Recovery: Log SMTP error. Do NOT reveal SMTP details to user.
  
  CURRENT ISSUE: src/app/api/auth/otp/send/route.ts sends email AFTER storing
  OTP in Firestore. If email send fails, OTP is stored but undelivered.
  Stale OTP document blocks future attempts for that email.
  [Recommendation] Store OTP AFTER successful email delivery, not before.

ERROR: RATE LIMIT EXCEEDED (429)
  Cause: Too many requests from a single source
  User-facing message: "Too many attempts. Please wait before trying again."
  Internal log: "RATE_LIMIT: endpoint={route} source={ip}"
  Severity: MEDIUM
  Recovery: Return 429 with Retry-After header. Do NOT reveal limit details.

ERROR: SERVER INTERNAL ERROR (500)
  Cause: Unhandled exception in API route
  User-facing message: "An unexpected error occurred. Our team has been notified."
  Internal log: Full error with stack trace to centralized logging service
  Severity: HIGH
  
  CURRENT ISSUE: error.tsx exposes raw error.message to the user.
  This can leak internal path names, function names, or stack frames.
  [Recommendation] Use a generic message in production. Log full error server-side.

15. ABUSE PREVENTION
================================================================================

RECOMMENDED RATE LIMITS
-------------------------
| Endpoint                              | Limit               | Window    |
|---------------------------------------|---------------------|-----------|
| POST /api/auth/otp/send               | 3 per email         | 10 min    |
| POST /api/auth/otp/verify             | 5 per email         | 10 min    |
| POST /api/rooms                       | 20 per recruiter    | 24 hours  |
| POST /api/rooms/[id]/candidate        | 3 per room          | 24 hours  |
| POST /api/rooms/[id]/alerts           | 100 per room        | Per session|
| POST /api/rooms/[id]/trust            | 1 per 10 seconds    | Per room   |
| POST /api/rooms/[id]/session/start    | 1 per room          | Lifetime  |
| POST /api/rooms/[id]/session/end      | 2 per room          | Lifetime  |
| POST /api/rooms/[id]/report           | 3 per room          | 24 hours  |
| All API routes                        | 1000 per IP         | 1 hour    |

SPAM PROTECTION
  Risk: The OTP send endpoint can be used to spam any email address with
  GetAWay-branded OTP emails.
  [Recommendation] Verify email exists in Firebase Auth BEFORE sending OTP.
  This prevents the endpoint from being used as a spam tool for non-users.

BRUTE FORCE PROTECTION
  Risk: OTP verify endpoint — 6-digit code = 1,000,000 combinations.
  Without rate limiting, brute force takes seconds.
  [Recommendation] Implement per-email attempt counters in Redis/Vercel KV.
  Lock after 5 failed attempts. Send alert email on lockout.

OTP ABUSE
  Risks: 
  1. OTP flooding (send many OTPs to overwrite valid one)
  2. OTP replay (use same code twice)
  3. Race condition (two requests use same OTP simultaneously)
  
  Current State:
  - Replay prevented (used: true flag)
  - Flooding not prevented (new send overwrites old OTP document)
  - Race condition possible (no atomic transaction on verify+mark-used)
  
  [Recommendation] Use Firestore transactions for OTP verify+mark-used.
  Add document TTL and rate limiting for OTP sends.

ROOM ID ENUMERATION
  Risk: 9,000 possible room IDs — exhaustive scanning in minutes.
  [Recommendation] Change to 128-bit random room IDs.
  Consider returning a generic 404 (not "Room not found") to avoid confirming
  that room IDs exist.

CREDENTIAL STUFFING
  Risk: Attackers use leaked email/password lists against Firebase Auth login.
  Firebase Auth has built-in brute-force protection but it can be tuned.
  [Recommendation] Enable Firebase Auth's advanced security features:
  - reCAPTCHA Enterprise integration
  - App Check for Firebase (blocks unauthorized clients)
  - Identity Platform for enterprise-grade login protection

API ABUSE (General)
  Risk: No API abuse prevention exists across any endpoint.
  [Recommendation]:
  1. Implement Vercel middleware with rate limiting (e.g., @upstash/ratelimit)
  2. Add Firebase App Check to require valid client certificates
  3. Monitor for anomalous request patterns via Vercel analytics

DENIAL OF SERVICE
  Risk: Large volume of WebRTC session creation, Firestore writes to open
  subcollections, or SMTP OTP spam could degrade service.
  [Recommendation]:
  1. Set Firestore document write quotas per room
  2. Implement WAF (Web Application Firewall) via Cloudflare if using custom domain
  3. Enable Vercel protection rules for DDoS mitigation

17. EDGE CASES & FAILURE SCENARIOS
================================================================================

SCENARIO 1: CANDIDATE CLOSES TAB MID-INTERVIEW
  Expected Behavior: WebRTC connection drops. Interviewer sees "disconnected" state.
  Current Implementation: hangup() called on component unmount. cleanupSignal()
    NOT called by candidate (only called by interviewer on leave).
  Recovery Strategy: Interviewer should be able to share the link again. Candidate
    can rejoin if room is still ACTIVE. WebRTC renegotiation should restart.
  User Experience Gap: No alert to interviewer that candidate disconnected.
  [Recommendation] Use Firestore presence to detect candidate disconnection.
    Show "Candidate disconnected" banner in InterviewerView.

SCENARIO 2: RECRUITER DISCONNECTS (INTERVIEWER LEAVES)
  Expected Behavior: WebRTC connection drops. Candidate sees "failed" state.
  Current Implementation: cleanupSignal() called on leave. Candidate's useWebRTC
    detects disconnection via onconnectionstatechange → "failed" state.
  Recovery Strategy: LiveRoomStep shows "Retry" button.
  Gap: Session state in Firestore remains "ACTIVE" — not automatically ended.
  [Recommendation] Detect interviewer disconnect server-side (e.g., via presence
    system) and auto-end the session.

SCENARIO 3: WEBRTC NEGOTIATION FAILS (ICE FAILURE)
  Expected Behavior: useWebRTC sets connectionState to "failed".
  Current Implementation: LiveRoomStep and InterviewerView show error UI with Retry.
  Recovery Strategy: User clicks Retry → window.location.reload().
  Gap: Stale ICE candidates in Firestore may cause re-negotiation to fail again.
  [Recommendation] On retry, trigger cleanupSignal() before restarting.

SCENARIO 4: FIRESTORE UNAVAILABLE
  Expected Behavior: All API routes return 503. WebRTC signaling fails silently.
  Current Implementation: No explicit Firestore outage handling. Errors caught by
    API try/catch and return 500.
  User Experience: Candidate or recruiter sees "Something went wrong" error page.
  [Recommendation] Detect Firestore unavailability and show "Service temporarily
    unavailable" with status page link. Implement Firestore retry logic.

SCENARIO 5: DUPLICATE OTP REQUESTS (Race Condition)
  Expected Behavior: One OTP is valid; others are overwritten.
  Current Implementation: Each OTP send overwrites otpCodes/{email}. If two
    requests arrive simultaneously, the last write wins. The email with the
    first OTP becomes invalid.
  Recovery Strategy: User receives second email with valid OTP. First email's
    code will fail with "Invalid OTP."
  [Recommendation] Use Firestore transactions for OTP creation. Reject new OTP
    if one is already active and unexpired.

SCENARIO 6: EXPIRED ROOM LINK
  Expected Behavior: Candidate with old link should see "Session ended" page.
  Current Implementation: No room status check at page load. Candidate can
    proceed through onboarding even if room.status == "ENDED."
  [Recommendation] Check room status in CandidateFlow on mount. If status is
    "ENDED," redirect to a "Session has ended" page.

SCENARIO 7: DELETED ROOM ACCESS
  Expected Behavior: 404 page for invalid room IDs.
  Current Implementation: DELETE route requires auth (correct). But if room
    doesn't exist, /api/rooms/[id]/candidate returns 404. Interview page
    doesn't check for room existence at page load — it renders the onboarding
    form regardless.
  [Recommendation] Server-side check room existence before rendering interview
    page. Return 404 with "This interview room no longer exists."

SCENARIO 8: MULTIPLE CANDIDATES JOIN SAME ROOM
  Expected Behavior: Only one candidate should be in a room at a time.
  Current Implementation: No enforcement. Multiple browsers can go through
    CandidateFlow for the same room ID.
  Impact: WebRTC signaling gets corrupted (multiple parties write to same
    signal/data document). The later joiner overwrites the earlier one.
  [Recommendation] Check rooms.candidate field at onboarding submit. If
    candidate != null, reject with "A candidate has already joined this room."

SCENARIO 9: OTP SEND SUCCEEDS BUT EMAIL DELIVERY FAILS
  Expected Behavior: Graceful failure with retry option.
  Current Implementation: OTP is stored in Firestore BEFORE email is sent.
    If email send fails, OTP document exists but email was never delivered.
    Next OTP request overwrites the document — the original OTP is inaccessible.
  [Recommendation] Reverse the order: attempt email send first. Only store
    OTP document in Firestore after successful email delivery.

SCENARIO 10: TRUST SCORE GOES TO 0 DURING SESSION
  Expected Behavior: Alert fires, recruiter is notified.
  Current Implementation: If score < recruiter threshold, an alert is created.
    No special handling for score == 0 (complete loss of trust).
  [Recommendation] Add a configurable "session termination threshold." If trust
    score drops to 0 (or configurable floor), automatically end session and
    notify recruiter via webhook.

SCENARIO 11: WEBHOOK DELIVERY FAILURE
  Expected Behavior: Session end is not blocked by webhook failure.
  Current Implementation: Webhook is fire-and-forget (.catch(() => {})). Session
    end always returns success regardless of webhook outcome.
  Gap: Recruiter has no visibility into webhook delivery failure.
  [Recommendation] Log webhook delivery outcomes. Implement retry with backoff.
    Notify recruiter in dashboard if webhook consistently fails.

SCENARIO 12: INTERVIEW PAGE ACCESSED WITHOUT ROOM ID
  Expected Behavior: Redirect or error page.
  Current Implementation: Next.js dynamic route (/interview/[roomId]) requires
    roomId parameter. If navigated to /interview/ without ID, Next.js returns 404.
  Assessment: HANDLED by Next.js routing.

20. FINAL SECURITY ASSESSMENT

PRODUCTION READINESS SCORE: 3.5 / 10

This score reflects that GetAWay has a functional, well-designed architecture
with correct security for recruiter-facing data, but has significant gaps in
the candidate/session API layer that make it unsuitable for real hiring
workflows without immediate remediation.

MAJOR BLOCKERS (Must fix before real users)
--------------------------------------------
1. WEAK ROOM IDS: 9,000 possible values — enumerable in 90 seconds.
   All session security depends on room ID confidentiality.

2. NO API AUTHENTICATION ON SESSION WRITES: Trust scores, alerts, and
   candidate data can be written by anyone on the internet.

3. NO RECRUITER DASHBOARD PROTECTION: Any URL visitor can access
   /recruiter/overview without authentication (UI only protection).

4. INTERVIEWER VIEW IS PUBLIC: Any person with a room ID can claim
   the interviewer role by adding ?view=interviewer.

5. NO OTP RATE LIMITING: Account takeover via brute force is trivial.

6. MOCK STORE IN PRODUCTION PATH: Data is lost on browser refresh
   across devices — the application cannot function reliably for real sessions.

TOP 10 SECURITY IMPROVEMENTS (Priority Order)
----------------------------------------------
1. Implement Next.js middleware.ts with Firebase token verification for
   all /recruiter/* routes. This single change eliminates unauthorized
   recruiter dashboard access.

2. Change room ID generation to UUID v4 / 128-bit random hex.
   This eliminates the room enumeration attack surface.

3. Add authentication to session-write API routes (alerts, trust,
   candidate, session start/end). Use session tokens or JWT for
   unauthenticated participants.

4. Add rate limiting to OTP endpoints (send: 3/10min, verify: 5/10min).
   Implement account lockout after 5 failed OTP attempts.

5. Lock down Firestore subcollections. Change alerts, trustLogs, report,
   and webrtc from allow read, write: if true to server-only writes.

6. Fix crypto: Replace Math.random() OTP generation with
   crypto.randomBytes() for cryptographically secure OTPs.

7. Add HTTP security headers (CSP, HSTS, X-Frame-Options) in next.config.ts
   to prevent clickjacking, content injection, and MITM attacks.

8. Replace mockStore with real Firestore API calls in all recruiter pages
   so that session data persists reliably across devices and sessions.

9. Validate and sanitize webhook URL on write to prevent SSRF.
   Block RFC1918 addresses in the webhook URL field.

10. Replace public TURN server (openrelay.metered.ca) with a private,
    authenticated TURN server to protect media relay traffic.

CLOSING NOTE FOR FOUNDERS AND DEVELOPERS
-----------------------------------------
GetAWay has an excellent foundation. The Firebase + Next.js + WebRTC
architecture is sound. The authentication model for recruiters is correctly

The gaps identified in this document are primarily around the candidate-facing
and session APIs, which were built with the intentional design choice of
"no authentication for candidates to reduce friction." This is a valid product
decision for a final-year project, but it creates real security risks for
production deployment.

The good news: most critical issues can be fixed in 1-2 weeks of focused
engineering work, primarily by:
  1. Adding middleware
  2. Increasing room ID entropy
  3. Adding rate limiting
  4. Locking Firestore rules

Once these four items are addressed, GetAWay will be significantly more secure
and suitable for real-world use in controlled hiring environments.
