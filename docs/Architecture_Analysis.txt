Executive Summary
GetAWay is a browser-native, AI-powered interview proctoring platform built on a modern, serverless tech stack. The application follows a clean separation of concerns with a Next.js 16 frontend, real-time Firestore backend, and WebRTC P2P video infrastructure.
Key Characteristics:
Architecture Type: Monolithic Next.js application with serverless API routes
Deployment Model: Vercel (serverless) + Firebase (managed services)
State Management: Hybrid (Zustand client-side + Firestore real-time listeners)
Real-time Capability: Firestore listeners for alerts, trust scores, and WebRTC signaling
Authentication: Firebase Auth (email/password + OTP via SMTP)
Video Protocol: WebRTC with Firestore-based signaling (no separate signaling server)
Critical Finding: v1 uses an in-memory mock store (mockStore.ts) that doesn't persist across server restarts, making it unsuitable for production beyond MVP testing.

Component Lifecycle & Data Flow
Recruiter Flow
1. User → /recruiter/overview
   ↓
2. RecruiterLayout renders sidebar + outlet
   ↓
3. OverviewPage queries mockStore.rooms
   ↓
4. Renders KPI cards (total interviews, candidates, avg duration)
   ↓
5. User clicks "Initialize" → /recruiter/initialize
   ↓
6. InitializePage renders form
   ↓
7. On submit → mockStore.createRoom() → generates room ID
   ↓
8. Returns candidate + interviewer links
   ↓
9. User shares links or copies to clipboard

Candidate Flow
1. User opens invitation link → /interview/[roomId]
   ↓
2. Page renders CandidateFlow (multi-step component)
   ↓
3. Step 1: OnboardingStep
   - User fills name, email, role
   - On submit → mockStore.updateCandidate(roomId, data)
   - Transitions to Step 2
   ↓
4. Step 2: RulesStep
   - Displays consent checkbox
   - On confirm → Transitions to Step 3
   ↓
5. Step 3: LiveRoomStep
   - useWebRTC hook initializes
   - Requests browser camera + microphone
   - Negotiates offer/answer via Firestore /webrtc/{roomId}/signal
   - On connection → displays video feed
   ↓
6. Video call active
   - Real-time alerts subscribed via Firestore listeners
   - Trust score updated via POST /api/rooms/[id]/trust
   ↓
7. Interview ends → User redirected to /thankyou

Interviewer Flow
1. User joins via link with ?view=interviewer
   ↓
2. Page renders InterviewerView (live monitoring panel)
   ↓
3. useWebRTC hook creates offer, sends to Firestore
   ↓
4. Waits for candidate to join (answer)
   ↓
5. Displays remote video + local PiP
   ↓
6. Recruiter can:
   - Toggle mic/camera
   - Share screen (via getDisplayMedia API)
   - Issue manual warnings
   - Monitor trust score (mock implementation)
   ↓
7. On "Leave Room" → Call hangup(), cleanup Firestore signal data
   ↓
8. Redirected to /recruiter/sessions

Backend Architecture
API Route Structure & Responsibilities
Authentication Routes
File: src/app/api/auth/
POST /api/auth/otp/send
Purpose: Generate 6-digit OTP, hash with SHA-256, store in Firestore
Responsibility: Email OTP via Nodemailer
Security: Server-side hashing; expiry set to 10 minutes
Data Store: otpCodes/{email} (server-only collection)
POST /api/auth/otp/verify
Purpose: Verify OTP, reset recruiter password
Responsibility: Validate hash, mark OTP as used, update Firebase Auth
Security: One-time use enforcement; expiry check; hash comparison
Response: Custom Firebase token for immediate login
Room Management Routes
File: src/app/api/rooms/
POST /api/rooms
Purpose: Create interview room
Auth: Requires Firebase ID token
Responsibility: Generate unique room ID, store room doc in Firestore
Data Store: rooms/{roomId} with recruiter FK
GET /api/rooms
Purpose: List recruiter's rooms
Auth: Requires Firebase ID token
Query: recruiterId == auth.uid, orderBy createdAt DESC
POST /api/rooms/[roomId]/candidate
Purpose: Save candidate onboarding data
Auth: No auth (room ID is unguessable)
Responsibility: Validate input (Zod schema), update room doc with candidate info
Data Store: rooms/{roomId} field candidate
POST /api/rooms/[roomId]/trust
Purpose: Log trust score event
Auth: None (room ID is unguessable)
Responsibility: Append to trustLogs subcollection; auto-alert if below threshold
Side Effect: If score < recruiter.settings.aiThreshold, create alert
POST /api/rooms/[roomId]/alerts
Purpose: Create alert (AI or manual)
Auth: None (room ID is unguessable)
Responsibility: Append to alerts subcollection
Fields: message, issuedBy (AI|INTERVIEWER), severity, category, metadata, createdAt
POST /api/rooms/[roomId]/session/start
Purpose: Mark session as ACTIVE
Responsibility: Update session.status and session.startedAt
POST /api/rooms/[roomId]/session/end
Purpose: Mark session as ENDED
Responsibility: Update session.endedAt; optionally fire webhook
POST /api/rooms/[roomId]/report/compile
Purpose: Compile post-session report
Responsibility: Fetch all trustLogs + alerts; build timeline; write to report subcollection
Analytics Routes
File: src/app/api/analytics/
GET /api/analytics?days=7
Purpose: Return recruiter's session trends
Auth: Requires Firebase ID token
Query: Fetch rooms by recruiter + createdAt range; group by date
Response: Array of {date, name, sessions, avgTrust}
Settings Routes
File: src/app/api/settings/
GET /api/settings
Purpose: Fetch recruiter preferences
Auth: Requires Firebase ID token
Data Store: recruiters/{uid} field settings
PUT /api/settings
Purpose: Update recruiter preferences
Auth: Requires Firebase ID token
Validation: Zod SettingsSchema
Data Store: Merge into recruiters/{uid} field settings
API Security Model
Authentication Method:
Firebase Client SDK generates ID tokens on client
Token sent in Authorization: Bearer <token> header
Server verifies via Firebase Admin SDK: adminAuth.verifyIdToken(token)
On success: Extract uid from decoded token
Authorization Model:
Recruiter routes: Require authenticated uid (via verifyAuthToken)
Room routes: Use "security by obscurity" (room ID is unguessable GW-XXXX)
Firestore rules: Mirror API-level checks (defense in depth)
Error Handling:
// lib/api.ts
export function apiSuccess<T>(data: T, status = 200) {
  return NextResponse.json({ success: true, data }, { status });
}

export function apiError(message: string, status = 400) {
  return NextResponse.json({ success: false, error: message }, { status });
}

export function apiValidationError(error: ZodError) {
  return NextResponse.json(
    {
      success: false,
      error: "Validation failed",
      details: error.flatten().fieldErrors,
    },
    { status: 422 }
  );
}

Authentication Architecture
Identity Flow
Recruiter Authentication:
1. User visits /recruiter/overview (redirect from home page)
2. Firebase Auth checks if user logged in
   - If not → Show login/signup form
   - If yes → Show dashboard
3. On signup/signin → Firebase generates ID token
4. Token stored in Firebase SDK (automatic)
5. All API calls include token in Authorization header
6. Server verifies token via adminAuth.verifyIdToken()
7. On success → Extract uid for data isolation

Candidate Authentication:
1. Candidate receives invitation link: /interview/[roomId]
2. NO login required
3. Room ID acts as the credential (unguessable: GW-XXXX format)
4. Onboarding form collects name, email, role (for audit trail)
5. No password needed
6. Consent checkbox required

OTP Flow (Password Reset):
1. Recruiter clicks "Forgot Password"
2. Enters email → POST /api/auth/otp/send
3. Server generates 6-digit OTP
4. Hash OTP with SHA-256 (NEVER store plain text)
5. Store hash in Firestore: otpCodes/{email}
6. Send plain OTP via email (Nodemailer SMTP)
7. Recruiter receives email with 6-digit code
8. Enters OTP + new password → POST /api/auth/otp/verify
9. Server verifies hash matches
10. Check if OTP expired (10 minutes)
11. Mark OTP as used (prevent replay)
12. Update Firebase Auth password
13. Return custom token
14. Client logs in immediately

Firebase Admin SDK Role:
Verifies ID tokens server-side
Creates custom tokens for OTP password resets
Updates password in Firebase Auth
Hashes OTP (never client-side)
Trusted identity source
Critical Security Detail:
OTP stored as SHA-256 hash, NOT plain text
File: src/app/api/auth/otp/send/route.ts line 19
Reason: If Firestore is ever compromised, OTPs can't be used

Firestore Security Rules
Current Rules (v1):
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /recruiters/{uid} {
      allow read, write: if request.auth != null && request.auth.uid == uid;
    }
    
    match /rooms/{roomId} {
      allow read: if true;                              // Anyone with ID
      allow create: if request.auth != null;           // Auth required
      allow update: if true;                           // Candidates save info
      allow delete: if request.auth.uid == resource.data.recruiterId;
      
      match /alerts/{alertId} {
        allow read, write: if true;
      }
      match /trustLogs/{logId} {
        allow read, write: if true;
      }
      match /report/{doc} {
        allow read, write: if true;
      }
    }
    
    match /webrtc/{roomId} {
      match /signal/{doc} {
        allow read, write: if true;
      }
      match /callerCandidates/{doc} {
        allow read, write: if true;
      }
      match /calleeCandidates/{doc} {
        allow read, write: if true;
      }
    }
    
    match /otpCodes/{email} {
      allow read, write: if false;  // Server-only
    }
  }
}

Security Philosophy:
Recruiter data: Scoped by request.auth.uid
Room data: Protected by unguessable room ID (no auth needed)
WebRTC signaling: Fully open (room ID is unguessable cryptographic value)
OTP codes: Completely blocked on client (server-only via Admin SDK)
Risk Analysis:
Risk: If someone guesses a room ID, they can read/write alerts
Mitigations: Room ID format (GW-XXXX) is 6.4 billion possibilities; random seed prevents patterns
Assumption: Room IDs are treated as secrets (not shareable with untrusted parties)

Database Architecture
Firestore Collections & Subcollections
Collection: recruiters
Document ID: Firebase Auth UID
Purpose: Recruiter profile, settings, subscription
Fields: email, displayName, company, subscription, settings, timestamps
Access: request.auth.uid == {uid}
Indexes: Email lookup (for OTP)
Collection: rooms
Document ID: Unique room ID (GW-XXXX format)
Purpose: Interview session metadata
FK: recruiterId (links to recruiter)
Subcollections: alerts, trustLogs, report, metrics
Access: No auth (room ID is unguessable)
Queries: recruiterId == ? && createdAt DESC (requires index)
Subcollection: rooms/{roomId}/alerts
Purpose: AI-detected or manual warnings
Fields: message, issuedBy, severity, category, metadata, createdAt
Access: Open read/write (room ID scoped)
Timeline: Used for post-session report
Subcollection: rooms/{roomId}/trustLogs
Purpose: Immutable audit trail of trust score changes
Fields: score, delta, reason, triggeredBy, createdAt
Access: Open read/write (room ID scoped)
Immutability: Never updated (only appended)
Query: createdAt ASC for chronological reports
Subcollection: rooms/{roomId}/report
Purpose: Compiled post-session analysis
Fields: candidateName, candidateRole, finalTrustScore, alertCount, timeline, compiledAt
Access: Open read/write (room ID scoped)
Uniqueness: Single document per room (ID: "compiled")
Collection: webrtc
Purpose: WebRTC signaling data (separate from rooms for security)
Subcollections: signal, callerCandidates, calleeCandidates
Access: Fully open (room ID is unguessable)
Reason for separate collection: Prevents auth rule conflicts
Subcollection: webrtc/{roomId}/signal
Document ID: "data" (one per room)
Fields: offer, answer, createdAt, expiresAt
TTL: Auto-delete after 1 hour (Firestore TTL policy)
Purpose: SDP offer/answer exchange
Subcollection: webrtc/{roomId}/callerCandidates
Purpose: Interviewer's ICE candidates
Fields: candidate (ICE candidate string), sdpMLineIndex, sdpMid, createdAt
Append-only
Subcollection: webrtc/{roomId}/calleeCandidates
Purpose: Candidate's ICE candidates
Fields: Same as callerCandidates
Append-only
Collection: otpCodes (Server-Only)
Document ID: Email address
Purpose: OTP verification (password reset)
Fields: hash (SHA-256), expiresAt, used, createdAt
Access: allow read, write: if false (server-only via Admin SDK)
Cleanup: Auto-delete after 10 minutes (Firestore TTL)
Collection: analytics (Future)

Failure Scenarios & Recovery
Scenario 1: Network drops during call
Handler: oniceconnectionstatechange → "disconnected"
Current behavior: Set state to "failed" → Show retry button
Ideal behavior: Auto-reconnect, exponential backoff
Scenario 2: Interviewer leaves, candidate still waiting
Handler: Interviewer calls hangup() → clears Firestore signal data
Candidate effect: No remote stream; shows "Waiting for interviewer..."
Ideal: Timeout after 5 minutes → redirect to thank you page
Scenario 3: Candidate slow/poor network
Symptom: Audio/video very delayed
Cause: TURN relay being used (low bandwidth)
Mitigation: Firestore rules / WebRTC constraints could limit quality

File: src/app/api/rooms/[roomId]/report/route.ts
Endpoint: POST /api/rooms/[roomId]/report/compile
Process:
1. Fetch room by ID
2. Fetch all trustLogs subcollection (orderBy createdAt ASC)
3. Fetch all alerts subcollection (orderBy createdAt ASC)
4. Get final trust score (last trustLog entry)
5. Build timeline:
   - Start event: "Candidate joined"
   - Alert events: Each alert with timestamp
   - End event: "Session ended" (if endedAt exists)

7. Compile report object:

     roomId,

     timeline,
